Understanding Git and GitHub
What is Git?

Git is a distributed version control system

  • A version control system is a system that tracks and manages changes to folders/files.
  • A distributed version control system takes that same system and adds a peer-to-peer model to it.

Git is NOT the same as Github. Github is a platform built on top of Git.

Why use Git?
  • Track every change: See the complete history of your project and easily revert to previous versions.
  • Collaborate seamlessly: Work on the same project with others without conflicts or messy merging.
  • Experiment without risk: Use branches to try out new features without affecting the main codebase.
  • Recover from disasters: Never lose your work again with Git's complete version control.

Initialize a Repository

  • To create the .git directory

    • Git Directory

    • The .git directory contains all of the git data, from the upstream repository to change history.
    • This is what makes a repository a repository

    Staging Changes

    Staging is the process of adding files to be tracked.

    • To stage changes, use the command git add <filename>
      • Multiple files and directories can be separated by spaces
    • To stage all changes, use the command git add .

    Committing Changes

    Committing is the process of saving those staged changes into an object, which can later be pushed.

    • To commit staged changes, use the command git commit -m "<message>"

    Branching

    Allows multiple “save points” for your work.
    • New features can be created a new branch, which is later merged into the main branch
    • If issues arise on one branch, it does NOT impact any other branch
    • To update the remote repo, push changes (git push)

    Remote Repository

    A remote repository is the version of the project, which is hosted on a network.

    • Remote repositories can be hosted on a variety of platforms, such as Github, Gitlab, Bitbucket, etc.
    • GitHub is the most popular
  • There can be multiple remotes

    • Pull Changes

    Pulling is the process of updating your local repository with the changes from the remote repository.

    • To pull changes, use the command git pull

    Tags

    Tags are used to mark a specific point in the project's history.

    Tags vs. Branches

    • Tags are used to mark a specific point in the project's history.
    • Branches are used to develop features, and are later merged into the main branch.

    Tags are frequently used to mark versions in a project.

    Creating a tag will generate a new file in .git/refs/tags

    Walkthrough: Initialize Repository

    On your local machine, initialize a repository.

    In project root, run the following command:

    
						git init

    Here is what that generated:

    
						.git/
						├── branches/    # Stores information about local branches
						├── config       # Contains configuration settings for the repository
						├── description  # Optional file for a short description of the repository
						├── HEAD         # Points to the current branch
						├── hooks/       # Contains scripts for custom Git actions
						├── index        # Tracks the state of the working tree
						├── info/        # Contains additional information about the repository
						├── objects/     # Stores all of the Git objects (commits, trees, blobs)
						└── refs/        # Contains references to commits and tags

    Remember that this is a peer-to-peer system

    We just initialized our local repository

    Walkthrough: Staging and Committing

    Let's create a file to track

    In project directory run the following command:

    
						touch index.html

    Now let's stage that file

    Run the following command:

    
						git add index.html

    If you have more files to stage, you can stage all of them at once by running this:

    
						git add .

    Let's see what we staged

    In project root, run the following command:

    
						git status

    Here is what you should see:

    
						On branch main

						No commits yet

						Changes to be committed:
							(use "git rm --cached <file>..." to unstage)
									new file:   index.html

    Notice that the file is staged, but not committed

    In project root, run the following command:

    
						git commit -m "Initial commit"

    You should see something similar to this:

    
						[main (root-commit) 5aedcb9] initial commit
						1 file changed, 0 insertions(+), 0 deletions(-)
						create mode 100644 index.html

    Let's break that down...

    [main (root-commit) 5aedcb9]

    • [main]: Indicates that the commit was made on the main branch, which is the default branch in many Git repositories.
    • (root-commit): Specifies that this is the very first commit in the repository's history, meaning there are no previous commits to reference.
    • 5aedcb9]: The unique 40-character hexadecimal hash that identifies this specific commit. It serves as a "fingerprint" for this version of the repository.

    1 file changed, 0 insertions(+), 0 deletions(-):

    • 1 file changed: Indicates that one file was changed in this commit.
    • 0 insertions(+): Indicates that no lines were added to the file.
    • 0 deletions(-): Indicates that no lines were removed from the file.

    create mode 100644 index.html:

    • create mode 100644: Indicates that a new file was created.
    • index.html: The name of the file that was created.

    The changes have been staged and committed, but they are still only on our local machine.

    Walkthrough: Create Remote Repository

    Let's create a remote repository on Github

    Go to Github and navigate to your profile overview. Near the top, you should see this:

    Click the green button that says "New"

    Name your repository

    Don't worry, you can change it later!

    Once you name the repo and select an owner, click the "Create repository" button

    Now we need to connect our local repository to the remote repository

    After creating the repository, you should see this:

    Copy the link starting with "git@github.com"

    In this case, it would be "git@github.com:Cooksey99/git_demo.git"

    In your local project repository, run the following command: 

    
						# git remote add <origin> <url>
						
						git remote add origin git@github.com:Cooksey99/git_demo.git
    • origin: the name used to reference this remote repository (remember there can be multiple)
    • url: the url of the remote repository

    Our local repository now has a connection with the remote repository

    Inside of the .git/config file, we can see that

    												[core]
						repositoryformatversion = 0
						filemode = true
						bare = false
						logallrefupdates = true
						ignorecase = true
						precomposeunicode = true
						[remote "origin"]
						url = git@github.com:Cooksey99/git_demo.git
						fetch = +refs/heads/*:refs/remotes/origin/*

    Notice the remote "origin" section

    The command automatically generated that for us.
